# Buddy Competitive Readiness

Buddy should compete as a trust-first vibe building platform. The goal is not only to generate a website. The goal is to turn one client prompt into an app, bot, media packet, course, game, simulation, dashboard, test evidence, approval path, and handoff.

## Capability roadmap

- One-prompt app and bot flow: prompt intake, build plan, files, tests, and pull request packet.
- Live preview while building: preview route, logs, screenshots, and test status.
- One-click deploy: approved artifact, hosting target, rollback plan, and deployment gate.
- Built-in services: auth, database, storage, payments, email, webhooks, sandbox fixtures, and secret policy.
- Rollback and checkpoints: branch, commit, diff summary, restore plan, and test evidence.
- Visual editing: revise layout, copy, scenes, lessons, game screens, and assets from preview feedback.
- Voice input: spoken prompts with transcript, consent, and safe command review.
- Image-to-app and screenshot-to-dashboard: uploaded images become component maps, responsive UI, and tests.
- Eval loop after every change: schema, syntax, sandbox, storage, readiness, and dashboard checks.

## Media safety stance

Buddy should be stricter than a social media baseline. Voice and image cloning require written consent, labeling, asset provenance, and review. Buddy blocks impersonation, cloned minors, fake authority, fake testimonials, sexual or degrading fake media of real people, and automated outreach with cloned voices.

## Demo promise

For clients, Buddy should say: describe what you want, then Buddy builds the packet, prototype, test evidence, and review path. Anything risky stays blocked until the client and owner approve it.
